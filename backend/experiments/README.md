# HITL experiment harness

Shared scaffolding for the post-hoc-rating vs self-escalation experiment
described in `/Users/celin/.claude/plans/mossy-twirling-rabin.md`. This
directory lives on `experiment/hitl-base` and is inherited (unchanged) by
the two arm branches:

- `experiment/hitl-postreview` — Approach A
- `experiment/hitl-escalate`   — Approach B

## Layout

| File | Purpose |
|---|---|
| `seed_fixture.py`  | Script-inserts a synthetic Cosmos collection with N known defects + near-miss non-defects. Emits `ground_truth.json`. |
| `ground_truth.json`| Generated by `seed_fixture.py`. Maps each seeded defect to its canonical `(field, category)` tuple. |
| `oracle.py`        | Rule-based rater. Given a finding, returns `{label, latency_ms}` by consulting `ground_truth.json`. Replaces a live human rater for reproducibility. |
| `harness.py`       | CLI orchestrator. Runs `--runs N --seeds S1,S2,...` against `--arm {control,A,B}`, plays the oracle, writes per-run JSON to `results/<arm>/seed_<n>/run_<i>.json`. |
| `metrics.py`       | Precision / recall / F1 / token-cost / Brier-score helpers. Pure functions over the per-run JSON. |
| `summarize.py`     | Reads `results/` and emits `results/summary.md` with a per-arm × per-run table, plus a reliability plot (Arm B). |

## End-to-end (planned)

```bash
# one-time
python -m backend.experiments.seed_fixture --collection hitl_eval --docs 15000 --seed 42

# per arm (after checking out the arm branch + updating the submodule)
python -m backend.experiments.harness --arm control --runs 5 --seeds 42,43,44
python -m backend.experiments.harness --arm A       --runs 5 --seeds 42,43,44
python -m backend.experiments.harness --arm B       --runs 5 --seeds 42,43,44

python -m backend.experiments.summarize results/
```

## Status

Scaffolding only. Each module defines its public interface (dataclasses,
function signatures, CLI shape) but the bodies raise `NotImplementedError`.
Fill in the modules in this order:

1. `seed_fixture.py` — generates the fixture all other modules consume.
2. `oracle.py`       — small, pure, easy to test in isolation.
3. `harness.py`      — wires the existing `/argus/run` + `/argus/runs/{id}` endpoints to the oracle.
4. `metrics.py`      — read what `harness.py` writes; no live calls.
5. `summarize.py`    — last; needs the metrics shape locked in first.
